Agent Evaluation / Field Guide

An LLM grades your agent. And it can be gamed.

Agentic systems run long, multi-step trajectories that nobody wants to grade by hand, so we point an LLM judge at them. That judge carries every classic bias, picks up a few that only appear once there are tools, steps, and rewards in the loop, and often turns into the very reward your agent is trained to chase.

~8 min read By Aditya Panchal 3 sources
Opening statement

A single answer is easy to grade. A trajectory is not.

A chatbot turn is one answer you can skim. An agent run is a whole story: a plan, a sequence of tool calls, the observations that come back, sometimes a few sub-agents arguing among themselves, and finally an answer. Reading thousands of those by hand is a non-starter, so teams do the obvious thing and point an LLM at the transcript to score whether the task succeeded and whether the path made sense.

It mostly works. The same line of research that made LLM judges popular (the MT-Bench study) found a strong judge agreeing with human reviewers more than 80 percent of the time, about the rate at which two humans agree. Modern agent-eval stacks lean on exactly this: when there is no single gold path, they let an LLM judge rate a trajectory for coherence, efficiency, and goal-directedness.

Here is the catch, and the agent setting sharpens it. A trajectory is long, so the judge's attention frays in the middle. Tool calls are hard to verify, so the judge takes the agent's word that the search happened and returned what it claims. And when that judge's score becomes a reward signal, the agent stops optimizing the task and starts optimizing the judge.

An agent trained against a biased judge does not get better at the task. It gets better at being graded.

The good news is the same as ever: almost every one of these failures has a known counter-move, and several of them are specific to agents (verify tools deterministically, judge the route and not just the destination). This is a field guide to both halves: where the judge gets fooled in an agentic system, and what to do about it.

How a verdict is made

One run in, one number out, and three places it bends

An agent run plus a rubric goes to the judge. A score comes out, and that score becomes an eval metric or, more dangerously, a training reward. Bias can enter at the run, at the read, and at the reward.

The run can be padded, the read can be fooled, and the reward can be hacked. A score that began as "did it work?" ends up steering training.
The charges

Twelve ways the bench gets it wrong

The first four are the classic LLM-judge biases, wearing agent clothes. The next eight are where the agent setting bites hardest: long runs, unverifiable tools, and a score that doubles as a reward.

Group 01

The classics, now with tools

Old biases that followed us into the agent era.

Position bias

When comparing two agent runs, the judge leans toward whichever it reads first, content aside.

Run A versus Run B can flip winners just by swapping which trajectory is pasted in first.

Fix: swap and average

Verbosity and step-count bias

Longer trajectories with more tool calls read as "more thorough," so a padded run beats an efficient one.

An agent that takes 12 wandering steps can outscore the one that nailed it in 3.

Fix: normalize length and steps

Style and confidence bias

A polished, confident final summary wins over a terse correct one. Narration gets graded as competence.

"Task completed successfully!" with headers beats a plain, correct one-liner.

Fix: score on a rubric

Self-preference and leakage

If the judge shares a model family with the agent backbone (it usually does), it favors its own kind of output.

One frontier model powering both the agent and the judge quietly inflates the agent's scores.

Fix: different judge family
Group 02

Where the agent setting bites

Failures that only exist because there are steps and tools.

Outcome-only blindness

The judge checks the final answer and waves the trajectory through, missing broken or illegal intermediate steps.

A correct answer reached by a forbidden action or a lucky guess scores like a clean solve.

Fix: judge the route too

Unverified tool use

The judge cannot see whether a tool truly ran or returned what the agent claims, so it trusts the narration.

An agent that says "I searched and found X" gets credit even if the call failed or never happened.

Fix: check tools deterministically

Long-trajectory fade

Long transcripts strain the judge's attention, so middle steps get skimmed (the lost-in-the-middle effect).

An error buried in step 7 of 15 slips past while the judge focuses on the start and the end.

Fix: chunk and judge in parts

Sycophancy to the goal

Hand the judge the intended outcome and it rubber-stamps any run that claims to have reached it.

Putting the target answer in the judge prompt turns scoring into confirmation.

Fix: hide the target
Group 03

When the judge becomes the reward

The training-time failures, which are the expensive ones.

Reward hacking

Optimize an agent directly against an LLM judge and it learns to please the judge instead of doing the task.

The policy discovers that confident success claims and verbose reasoning reliably earn points.

Fix: do not train on one judge

Self-evaluation loops

When an agent's own critic or reflection step uses the same model, it approves its own plans and inflates confidence.

The actor and the critic are the same model, so the review is mostly a formality.

Fix: separate actor and critic

Cascade trust

In multi-agent systems, one agent's output becomes another's ground truth, so an early biased verdict propagates.

A shaky judgment at step one is treated as fact by every downstream agent.

Fix: independent checks

Calibration drift

Swap the judge model, edit its prompt, or absorb a silent API bump and last month's scores stop being comparable.

Your success rate moves on the dashboard while no agent actually changed.

Fix: pin and anchor
The defense

Nine moves that hold up on appeal

You do not need a perfect judge. You need a judge whose blind spots you have measured and worked around. These run from "do it today" to "do it before you trust a number, or worse, train on it."

Judge the route, not just the destination

Score the trajectory step by step (tool choice, argument correctness, ordering, recovery from errors), not only the final answer. Most of the real signal lives in the path, and outcome-only scoring hides where the agent went wrong.

Cancels: outcome-only blindness

Verify tools deterministically, judge only the rest

Anything checkable should be checked by code, not vibes. Confirm a tool actually ran, with what arguments, and inspect the end state (did the record really get written?). Reserve the LLM judge for genuinely open-ended quality.

Cancels: unverified tool use, outcome-only blindness

Use a judge from a different model family

Never let the model powering your agent also grade it. Different lineage for the judge cuts self-preference and preference leakage; a panel from different providers is better still for high-stakes calls.

Cancels: self-preference, preference leakage

Swap, average, and anonymize

For pairwise run comparisons, evaluate both orderings and only count a win that holds both ways. Strip model names and identifying banners from the transcript before it reaches the judge.

Cancels: position, compassion-fade

Normalize for length and step count

Put it in the rubric explicitly: a concise, correct, 3-step run scores at least as well as a verbose 12-step one. Score per dimension on content, so polish and narration cannot win on their own.

Cancels: verbosity, step-count, style, confidence

Keep the target out of the judge prompt

Do not hand the judge the intended answer and ask "did it get here?" A judge that knows the destination will confirm it. Give it the rubric and the run, not the answer key.

Cancels: sycophancy to the goal

Chunk long runs before judging

Break a long trajectory into stages and judge each group, then aggregate, rather than dropping a 15-step transcript into one prompt and hoping the middle gets read. This fights the lost-in-the-middle fade.

Cancels: long-trajectory fade

Do not optimize against a single judge

If a judge becomes a training reward, the agent will hack it. Mix in deterministic, ground-truth reward where possible; hold out a separate judge for evaluation; add human spot-checks and adversarial probes that look for judge-pleasing behavior.

Cancels: reward hacking, self-evaluation loops

Pin the judge, anchor the runs, calibrate against humans

Freeze the judge version and prompt, re-run a fixed set of anchor trajectories every cycle to catch drift, and calibrate against human-labeled runs. Track reliability across repeated trials, not just an average, because an agent that succeeds sometimes is not the same as one you can ship.

Cancels: calibration drift, cascade trust, and the false confidence behind all of the above
Calibration is a loop, not a launch step. Re-run it whenever the judge, the rubric, or the agent's task distribution changes.
