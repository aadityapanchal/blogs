Agent Design / Field Guide

More tools should make a smarter agent. Loaded all at once, they make a worse one.

Every instinct says give the agent more tools and it can do more. But a model only chooses well from what it can see at once - load every tool into one context and selection accuracy slides toward random, while the definitions alone eat the window before the user says a word. Big systems genuinely need big catalogs; the mistake is showing the agent all of it at once. A field guide to tool sprawl: where the real ceiling is, and how to keep a large toolbox usable.

~7 min read By Aditya Panchal 3 sources
The toolbox

Adding a tool feels free. It isn't.

The obvious way to make an agent do more is to hand it more tools - another API, another integration, another MCP server. Capability looks additive: a hammer plus a saw plus a drill is strictly more than a hammer alone. So the toolbox grows, one reasonable addition at a time.

For an agent, it doesn't add up that way - because the cost isn't owning a tool, it's putting its definition in front of the model on every step. A model chooses well only from a short list it can hold in mind at once. Pile every tool into one context and, past some ceiling, the next tool stops expanding what the agent can do and starts shrinking what it can reliably reach. The capability is sitting right there in the box; the agent just can't find it.

The size of your catalog isn't the problem. How much of it the agent sees at once is.

This used to be a niche concern. Then MCP made the giant toolbox the default: connect a handful of servers - each exposing five to fifty tools - and your agent is suddenly choosing among hundreds, nearly all irrelevant to the task in front of it. A large system may genuinely need that many tools; the mistake is wiring them so the agent sees all of them on every call. It's a close relative of the gap in a 90% agent is a 35% product: capability you can demo, but can't depend on - except here it's capability you paid for and then buried.

When the box overflows

Four ways a full box turns on you

The failure isn't dramatic. The agent still shines in the demo, where it has a handful of tools. It degrades quietly as the visible list grows - in four compounding ways.

01

Selection collapse

Tool-selection accuracy holds while the visible list is short, then falls off non-linearly as it grows. Show the model enough at once and it's effectively guessing which one to call.

In the RAG-MCP benchmark, an agent facing a large tool set chose the right tool just 13.6% of the time - no better than random.

Measured: ~13.6% at scale, near chance
02

Semantic confusion

It isn't only about count. Tools with similar names or overlapping jobs interfere with each other, so the agent picks the wrong near-duplicate even when the right one is present.

Give it search_docs and search_files side by side and it flips a coin - or calls both and hopes.

Driver: similarity, not just size
03

The context tax

Every tool's schema sits in the context window on every call, before the user has typed a word. Cost and latency scale with the size of the toolbox - whether or not a tool is ever used.

Anthropic clocked 58 tools at ~55K tokens; Jira-style setups top 134K. Cloudflare's full API as tools would be 1.17M.

Tax: tens of thousands of tokens per call
04

Lost in the middle

A long tool list is a long context, and attention sags in its middle. Tools listed between the ends get quietly overlooked - even when one of them is exactly the right answer.

The correct tool is entry #200 of 400; the model never gives it serious consideration.

Position: middle tools get skipped
How full is the box

How full is too full?

The curve is the same across studies: fine while the list is short, then a cliff. This is about tools visible in one context - your catalog can be far larger, as long as the agent never sees all of it at once. Where you land tells you how much machinery you need.

Tools in one context What it takes to stay reliable
Under ~15 Plain function calling is fine - just keep names distinctthe model can hold them all at once
~15 - 50 Add a retrieval or embedding router; stop loading everything up frontthis is where the friction starts
~50 - 100 A dedicated selection layer - tool retrieval or a classifier - is no longer optionalflat lists become unreliable
100+ Progressive disclosure is mandatory; the agent should never see the full listflat loading is effectively broken

Exact numbers shift with the model and how distinct your tools are - a big, well-named catalog stretches further.
The rule that holds: keep the catalog as large as you need, the visible set small. The shape - smooth, then a cliff - does not change.

Only what's on the bench

Curate the bench, don't shrink the box

The goal isn't fewer capabilities - it's the bench, not the box. The whole catalog can stay in the box; only the handful of tools the agent needs right now belong out on the bench. Everything here keeps that working set small without shrinking the catalog.

Curate ruthlessly

Before any clever routing: delete tools. Merge near-duplicates, drop the ones used once a quarter, and give what's left sharp, non-overlapping names and descriptions. Most "too many tools" problems are really "too many redundant tools," and pruning costs nothing.

Cuts confusion and the context tax at the root

Show tools just in time

Don't load every schema up front. Give the agent a lightweight index plus a search_tools meta-tool, and load a tool's full definition only when it's about to be used. Anthropic's tool-search approach holds discovery to ~1.6-2.5K tokens whether the library has 40 tools or 400.

Keeps the working set tiny regardless of catalog size

Retrieve the right tools

Treat the catalog like a search index: embed the request, surface the top handful of relevant tools, and expose only those. This is what took RAG-MCP from 13.6% to 43% selection accuracy - same model, same tools, a smaller view.

Restores selection accuracy at scale

Route hierarchically

Split selection into coarse-then-fine: pick a category first, then a tool within it. It mirrors how a person reads a menu, and it sidesteps the flat-list cliff once your tools span several domains.

Beats flat selection across mixed domains

Call tools in code, not in context

Let the agent write code that invokes tools and passes results between them, instead of round-tripping every definition and payload through the prompt. Anthropic reports one task dropping from ~150K tokens to ~2K with this pattern.

Removes the context tax for multi-tool chains
